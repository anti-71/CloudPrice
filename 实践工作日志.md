## 第 1-2 天：环境搭建与项目初始化

### What

搭建 Python 开发环境，创建 GitHub 仓库，开通阿里云 OSS 对象存储与 ClickHouse 社区版实例。

### Why

**正例——为什么选阿里云 OSS 而不是本地文件存储：**

课程要求最终部署到云平台并通过 GitHub Actions 运行 CI/CD。如果选本地存储，个人电脑上的 CSV 文件无法被 GitHub Runner 访问，CI/CD 流完全跑不通。阿里云 OSS 与 ClickHouse 同地域（华东1杭州）部署，二者内网互通免流量费，且 OSS Python SDK (`oss2`) 仅 2 行代码即可完成上传。团队成员如果不同地点协作，本地文件需要人工传递，OSS 一个 bucket 所有人共享，减少同步成本。

**反例——第一次选择 ClickHouse 实例规格的错误：**

初始创建 ClickHouse 时，下意识选了 `4核16GB` 规格（"性能强肯定好"），运行一天后发现费用远超预期（每小时 4 元，每天接近 100 元）。复盘后意识到课程数据仅 2956 万行，理论单表扫描 ClickHouse 在 2核8GB 上也是毫秒级，根本不需要大规格。最终改为 `2核8GB`，性能完全够用，费用降至原来的 1/4。**关键教训：选型前先评估实际负载，不要习惯性"选最高配"。**

### How

- GitHub 仓库初始化、dev 分支策略
- 阿里云 OSS bucket `shiyuzhe` 创建，AccessKey 配置
- ClickHouse 社区版白名单、公网地址申请
- `.env` 配置密钥，`.gitignore` 排除，`.env.example` 提供模板

### 可复用经验

OSS + ClickHouse 同地域部署的方案：任何需要"文件存储 + SQL 分析"的项目都可复用此架构，关键是确保二者在同一 VPC/地域，否则会产生跨域流量费用。

## 第 3-4 天：数据生成与清洗

### What

编写模拟数据生成器（MockDataGenerator）和数据清洗模块（DataCleaner），实现 ODS → DWD 层的转换。

### Why

**正例——为什么从 Python dataclass 改为 pandas DataFrame：**

初版使用 Python `@dataclass` 对象表示每条记录，遍历列表进行清洗。测试发现生成 770 万行数据（1096 天）时，Python 对象占 1.5GB 内存，清洗耗时 25 分钟。更严重的是——无法处理课程下发的真实数据（2956 万行），直接 OOM 崩溃。

改为 pandas DataFrame 后：`df[df["price"] > 0]` 一条语句替代 10 行循环，底层 C 实现，内存占用降低 60%，同样 770 万行清洗从 25 分钟降到 2 分钟。2965 万行真实数据也能一次跑通。

**关键决策点：** 机器学习/数据分析项目，数据量 > 100 万行时直接用 pandas/numpy，不要手动写循环。这条经验在任何数据处理项目中都可复用。

### How

- 基于 pandas 重构 `mock_data_generator.py`：SKU 主数据预生成复用、每天 ±5% 价格波动、促销日 7-9 折
- `data_cleaner.py`：`df[df["price"] > 0]` 过滤负价、`drop_duplicates` 去重、中位数 10 倍过滤暴涨
- 脏数据注入：负价/负销量/空名/价格暴涨 4 种，注入率 2%

## 第 5-6 天：ClickHouse 连接与 SQL 计算

### What

连接阿里云 ClickHouse 社区版，将 OSS 数据导入 ClickHouse，用 SQL 实现清洗、汇总、指数计算。

### Why

**正例——为什么必须用 ClickHouse 而不是 Python 做计算：**

课程数据 2956 万行。Python pandas 虽然能加载，但做 `GROUP BY category_id + JOIN 类目权重 + 加权求和` 时，每步都是全量内存操作，单次计算 30 秒以上。ClickHouse 是列式存储引擎，同样的聚合查询 2 秒内完成，且支持分区（`PARTITION BY toYYYYMM(date)`）使得每日增量查询仅扫描当天的几十万行，不随时间增长而降速。

**排除 MySQL（反例）：**

试着在本地 MySQL 导入 100 万行数据跑同样的 GROUP BY 查询，耗时 12 秒。2956 万行预估 6 分钟以上，且 MySQL 不支持列存压缩，磁盘占用是 ClickHouse 的 3 倍。课程要求"高频日度指数计算"，MySQL 的 OLTP 架构不适合 OLAP 场景——这是两种完全不同类型的数据库，不能混用。

### How

- `clickhouse_client.py`：封装 HTTP 接口（GET 查询 / POST 插入 / TSV 格式批量导入）
- 建表：`ods_raw_prices` → `dwd_clean_prices` → `dws_*_daily_summary` → `ads_price_index_daily`
- 批量插入优化：100 行一批的 `INSERT VALUES` → 最终改为 TSV 格式 POST 上传（避免 URL 长度限制 404）
- 遇到坑：ClickHouse 白名单不放开导致 120 秒超时，多次排查后发现是阿里云安全组规则

### 可复用经验

ClickHouse HTTP 接口的 TSV 批量上传方案：当数据超过 1000 行时，不要用 `INSERT ... VALUES`（URL 长度限制），改用 `INSERT ... FORMAT TabSeparated` 配合 HTTP body 发送。这个经验在任何需要从 Python 批量写入 ClickHouse 的场景都适用。

## 第 7-8 天：可视化与 CI/CD

### What

用 matplotlib 生成日度价格指数折线图，配置 GitHub Actions 实现代码提交后自动测试+部署。

### Why

**为什么用 matplotlib 而不是 Quick BI：**

指导书推荐 Quick BI，但我选择了 matplotlib。原因是：

1. **本地调试效率**：matplotlib 一行 `plt.savefig()` 即可出图，Quick BI 需要上传数据 → 配置数据源 → 拖拽图表 → 发布仪表盘，调一个图的参数要绕 4 个页面。在"每天需要迭代图表样式"的课设节奏下，matplotlib 的反馈循环（改代码 → 2 秒出图）远快于 Quick BI（改配置 → 30 秒刷新）。

2. **可追溯**：matplotlib 图表的生成代码在 Git 里可追溯，Quick BI 的仪表盘配置是云端 JSON，无法版本控制。如果答辩时需要解释"这个颜色为什么是蓝色"，代码比截图更有说服力。

3. **如果约束变了**——如果课程要求搭建在线仪表盘（而非静态图片），那 Quick BI 是更好的选择，因为它天然支持 Web 嵌入和分享链接。

**为什么 CI 中 ClickHouse 连接需要 `continue-on-error: true`：**

GitHub Actions 的 Runner IP 可能不在 ClickHouse 白名单内（我们用的免费托管 Runner，IP 不固定），导致连接超时 120 秒。但单元测试和格式校验不依赖 ClickHouse，它们必须在 CI 里 100% 通过。所以架构设计为：ClickHouse 相关步骤设 `continue-on-error: true`，不阻塞核心 CI 流程。这是一个工程上的"优雅降级"决策。

### How

- `visualization.py`：ClickHouse SQL 查询 → matplotlib 双轴图（折线+涨跌柱状）
- `.github/workflows/ci.yml`：4 Stage 流水线（格式校验 → 单元+集成 → OSS验证 → 部署+图表）
- 24 个测试覆盖全部模块

## 第 9-10 天：真实数据接入与报告撰写

### What

接入课程下发的 CPI 真实数据集（1095 个 CSV，2956 万行），用 CPI 官方权重计算真实指数，生成最终折线图，完成设计报告和实践总结。

### Why

**反例——第一次用销量加权算指数导致结果异常：**

真实数据集没有 `sales`（销量）字段，只有 `weight`（CPI 权重）。最初没有仔细读 `products.csv` 的字段，误以为"没有销量就自己算销量"，结果指数从 100 飙到 1534，完全没有经济学意义——CPI 在 3 年内不可能涨 15 倍。

**正例——正确做法：**

仔细阅读 `categories.csv` 发现 `weight` 列（CPI 官方权重，比如"食品"权重 0.1869、"居住"权重 0.09），`products.csv` 也有每个商品在其类目下的权重。CPI 计算应该用这些官方权重做加权平均（类似拉氏指数公式），而不是淘宝式的"按销量加权"。

**如果约束变了**——如果未来数据包含真实的月销量，可以考虑将销量权重与 CPI 权重做复合加权（比如销量权重 × 0.3 + CPI 权重 × 0.7），但当前课程数据不具备这个条件。这个决策体现了"有什么数据用什么方法"的工程原则，而不是生搬硬套教材公式。

### How

- `process_real_data.py`：加载类目+商品权重 → 基期价格映射 → 逐天读 CSV → 类目加权 → 整体 CPI
- 2956 万行，2 分钟处理完毕（pandas 向量化）
- 写入 ClickHouse `cpi_index_daily` 表，生成 `cpi_index_trend.png`

### 可复用经验

**数据驱动的方法选择：** 先看数据有什么字段，再决定用什么算法——而不是反过来。这个教训在今后任何数据分析项目都是第一原则。
