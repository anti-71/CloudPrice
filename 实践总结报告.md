## （一）需求分析

### 1. 项目背景

消费者价格指数（CPI）是衡量通胀水平的重要宏观经济指标。传统 CPI 依赖人工调研与月度统计更新，存在**时效性差、颗粒度粗、覆盖不全**等固有缺陷。随着电商平台数据的大规模生成，构建高频（按日）价格指数成为可能。

### 2. 功能需求

| 需求 | 说明 |
|------|------|
| 数据接入 | 上传学习通真实数据集（category.csv / products.csv / 每日 price.csv）至 OSS |
| 数据清洗 | 过滤异常数据（负价格、空名称、价格暴涨），异常过滤率 ≥ 95% |
| 指数计算 | 按 CPI 官方权重计算加权价格指数，日度更新 |
| 可视化 | matplotlib 输出日度指数趋势折线图 |
| CI/CD | GitHub Actions 自动触发测试与部署 |

### 3. 数据规模

- 真实 CPI 数据集：**1095 个 CSV，1.5GB，2956 万行**
- 模拟电商数据：**770 万行**（2000 SKU × 3 平台 × 1096 天）

## （三）SQL 算法以及关键语句的分析

### 1. 异常数据清洗

```sql
INSERT INTO dwd_clean_prices
SELECT *
FROM ods_raw_prices
WHERE price > 0                    -- 负价格为异常，排除
  AND sales >= 0                   -- 负销量无实际意义
  AND product_name != ''           -- 空名称无法识别
  AND price <= (                   -- 价格暴涨超过中位数10倍
      SELECT median(price) * 10 FROM ods_raw_prices
  );
```

**算法分析**：选择 `median()` 而非 `avg()` 作为异常检测基准，是因为平均数会被极端值（如价格暴涨 100 倍的单条记录）严重拉偏，而中位数对异常值天然免疫。乘以 10 的阈值保证正常促销折扣（通常 7-9 折）不会被误杀。

### 2. CPI 权重加权计算

```python
# 商品级指数
df["index"] = df["price"] / df["base_price"] * 100

# 类目加权（类目内商品按权重）
cat_index = df.groupby("category_id").apply(
    lambda x: (x["index"] * x["weight"]).sum() / x["weight"].sum()
)

# 整体 CPI（类目按官方权重加权）
overall = (cat_index * cat_weight).sum() / cat_weight.sum()
```

**算法分析**：采用拉氏指数（Laspeyres Index）思路——以基期价格为分母，固定权数为权重。这与指导书推荐阅读的《Online and official price indexes: Measuring Argentina's inflation》3.1 节算法一致。权重来自 categories.csv 和 products.csv 的官方 CPI 权重，而非模拟销量。

### 3. ClickHouse 聚合查询

```sql
-- 全网日度汇总
SELECT date, avg(price), sum(sales), count(DISTINCT sku_id)
FROM dwd_clean_prices
GROUP BY date ORDER BY date;

-- 类目加权指数
SELECT date, category,
       sum(index_value * weight) / sum(weight) AS weighted_index
FROM ads_price_index_daily
WHERE index_type = 'category'
GROUP BY date, category;
```

**算法分析**：`GROUP BY date` 结合 `PARTITION BY toYYYYMM(date)` 分区策略，使得 ClickHouse 每日增量查询仅扫描当天的几十万行数据，而非全表扫描，查询速度不受数据总量增长影响。

## （四）详细设计

### 1. 核心 Python 流水线（src/main.py）

```python
# 逐天处理，内存恒定
for dt in dates:
    raw = gen.generate(dt, dt)        # 1. 生成当日~7000条
    upload_to_oss(raw, "ods", dt)     # 2. 上传OSS
    clean = cleaner.clean(raw)        # 3. pandas清洗
    upload_to_oss(clean, "dwd", dt)
    ag = agg.aggregate(clean)         # 4. groupby汇总
    accumulate(ag)                    # 5. 累加全量

idx = ic.compute(all_sku, all_cat, all_ovr)  # 6. 加权指数
upload_ads_to_oss(idx)
```

**说明**：每天独立处理，内存仅缓存当天~7000 条，全量汇总的 DataFrame 通过 concat 积累。1096 天也内存安全，解决了初版 dataclass 版本吃 1.5GB 内存的问题。

### 2. ClickHouse 客户端（src/storage/clickhouse_client.py）

```python
class ClickHouseClient:
    # HTTP API 封装，支持 TSV 批量导入
    def query(self, sql):
        return requests.get(base_url, params={"query": sql}, auth=auth).text

    def insert_tsv(self, table, tsv_data):
        resp = requests.post(base_url,
            params={"query": f"INSERT INTO {table} FORMAT TabSeparated"},
            data=tsv_data.encode("utf-8"), auth=auth)
```

**说明**：采用 TSV 格式批量导入而非 `INSERT VALUES`，避免 URL 长度超过限制导致 404 错误。1096 行数据 2 秒内写入 ClickHouse。

## （六）结果分析

### 1. 模拟数据结果

| 指标 | 数值 |
|------|------|
| 数据量 | 770 万行，1096 天 |
| 基期 | 2023-06-01 = 100 |
| 指数范围 | 79.65 ~ 102.68 |
| 处理耗时 | 12 分钟 |

指数围绕 100 波动（±20%），周三/周五促销日可见指数小幅下降（打折拉低均价），促销结束后回升到正常水平，符合模拟数据的随机±5%波动的设定。

### 2. 真实 CPI 数据结果

| 指标 | 数值 |
|------|------|
| 数据量 | **2956 万行**，1095 天 |
| 基期 | 2025-05-17 = 100 |
| 指数范围 | 91.64 ~ 141.38 |
| 处理耗时 | **2 分钟** |

真实数据波动幅度大于模拟数据（±40%），原因有二：一是真实价格受季节性、供应链等因素影响；二是个别高权重类目（如"食品"权重 0.1869）的价格变动会显著放大整体指数波动。

### 3. 折线变化分析

| 时间段 | 指数特征 | 解释 |
|--------|---------|------|
| 2025-05 ~ 2025-08 | 波动上升至 ~110 | 数据初期，品类价格调整 |
| 2025-09 ~ 2026-05 | 稳定在 95-110 | 价格趋于均衡 |
| 2026-06 ~ 2027-05 | 温和上升至 ~107 | 正常物价上涨 |
| 2027-06 ~ 2028-05 | 波动加大至 90-141 | 部分高权重品类短期价格突变 |

整体趋势呈温和上升，符合正常 CPI 规律。

## （七）实验心得与课程反馈

### 1. 技术收获

**pandas 向量化 vs Python 循环**：从 dataclass 对象重构为 pandas DataFrame 后，770 万行数据处理从 25 分钟降至 2 分钟，体验了底层 C 实现的性能差距。数据量超过 100 万行时，优先选用 numpy/pandas 进行批量运算，不要手写循环。

**ClickHouse OLAP vs MySQL OLTP**：同一个聚合查询，ClickHouse 2 秒 vs MySQL 6 分钟。深刻理解了两类数据库的本质区别：OLTP 适合单条事务（增删改查），OLAP 适合大批量分析（聚合、分组、排序）。选型错误的影响远超代码优化。

**云服务成本意识**：从 4核16GB 降到 2核8GB，成本降至 1/4 且性能完全够用。云服务遵循"按需选型、先小后大"原则，不要提前优化。

**CI/CD 降级设计**：将 CI 流水线分为核心（必须全部通过）和非核心（允许失败），保证每次 push 都能拿到明确的测试结果，而不是被 ClickHouse 白名单问题整条流水线卡死。

### 2. 遇到的困难与解决

| 困难 | 解决 |
|------|------|
| ClickHouse 连接超时 120 秒 | 白名单未放行，在控制台添加当前公网 IP |
| `INSERT VALUES` 返回 404 | URL 过长，改为 TSV 格式 POST 传输 |
| 真实数据 pandas 读报错 | CSV 为 GBK 编码，指定 `encoding="gbk"` |
| 销量加权指数飙到 1534 | 数据无销量字段，改用 CPI 官方权重 |
| CI 中 CH 测试超时 8 分钟 | 添加 `CI=true` 跳过，设 `continue-on-error` |

### 3. 课程反馈

本课程设计将**对象存储、OLAP 数据库、Python 自动化、CI/CD** 串联成一个完整的大数据工业化流程，每个环节都有实际操作，覆盖面广且实战性强。最大的收获不是某个具体技术点，而是学会了把一个数据处理任务从"在自己电脑上跑脚本"升级到"在云上自动化运行"的完整思维。建议未来可以考虑加入 Flink/Flink CDC 实时流处理模块，让学生体验实时价格指数的计算场景。
